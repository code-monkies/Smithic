# v0.2 — Autonomous Ideation

> **For the implementing agent.** This plan is self-contained. Read it cold, then read the current repo (`README.md`, `pyproject.toml`, and every file under `src/smithic/`) before changing anything. Do not assume any context outside what's written here and what's in the repo.

## Context

`v0.1` shipped the end-to-end pipeline (introspect → spec → implement → PR) with the feature description supplied via `--feature`. The interesting half — picking *which* feature to build based on real market signal — is what `v0.2` adds. After this release, `--feature` becomes optional and Smithic can run unattended against any repo.

Read the current state of the orchestrator at [src/smithic/orchestrator.py](../../src/smithic/orchestrator.py) to see how stages are wired today. The `research` and `score` stages already have placeholder slots in the config (see `ResearchConfig` and `RubricConfig` in [src/smithic/config.py](../../src/smithic/config.py)) but no implementation.

## Goals

A run invoked without `--feature` must:

1. **Research** the market based on the target's mission + introspection (web + Reddit), producing structured evidence.
2. **Score** several candidate features against a rubric, picking the highest-value one or aborting if none clear the threshold.
3. **Spec, implement** as today, but using the chosen candidate.
4. **Critique** its own diff before opening a PR; PR opens as draft if the critic flags concerns, or doesn't open at all if the critic says abort.
5. Stay inside its budget. The token ceiling has been hard since v0.1; tighten USD enforcement so it triggers mid-stage, not just between stages, in metered (`api`) mode.

A run invoked **with** `--feature` skips research+score and goes straight to spec+implement+critique+PR. Same shape as today plus the critique stage.

## Out of scope (do these in v0.3)

- Parallel runs / swarm coordination.
- Cross-run research cache (vector store).
- HN and Product Hunt MCP servers.
- Private-data signal sources (analytics, logs).
- A web dashboard.

If you find yourself building any of the above, stop and re-read this section.

## Architecture overview

The pipeline becomes:

```
introspect → research → score → spec → implement → critique → pr
                ↑          ↑                          ↑
              new        new                        new
```

When `--feature` is supplied, `research` and `score` are skipped and the supplied feature flows straight into `spec`.

When the critic returns `abort`, the run ends without a PR (worktree preserved for inspection). When the critic returns `revise`, the implement stage runs once more with critic feedback prepended; second failure → abort. When it returns `pass-with-concerns`, the PR is opened as a draft with a `smithic-needs-review` label.

## Decisions (settled — do not re-litigate)

| Question | Answer | Why |
|---|---|---|
| Research-source MCP for web | Tavily (off-the-shelf, agent-tuned) | Best signal-to-noise of the available search APIs; explicit support in the agent ecosystem. If the user has no Tavily key, fall back to `mcp-server-fetch` over a plain query URL — degrade gracefully, don't hard-fail. |
| Reddit MCP | Custom in-tree, public JSON API | Reddit's public `*.json` endpoints work without OAuth for read-only browsing. Don't require users to register an app for v0.2. |
| Score subagent | A dedicated Claude session with `output_format={"type": "json_schema", ...}` | Force structured output; one retry on parse failure; abort if still bad. |
| Critic subagent | Fresh Claude session with **no** implementation context, fed only the spec + the diff (`git diff <base>...HEAD`) | Independent review is the whole point. Sharing context defeats it. |
| Revise loop | Maximum 1 retry. Second failure → abort. | Spiral protection. The 41–86.7% production failure rate of multi-agent systems is dominated by unbounded loops. |
| USD enforcement tightening | Pre-call estimate based on prior stages' average + post-call recheck. Polled flag the orchestrator checks between substages. | Don't try to interrupt mid-call — that's hard and fragile. Catch breach at the next checkpoint. |
| Rubric format | YAML, loaded at run start, validated against a Pydantic schema | Already started in [`src/smithic/rubric/default.yaml`](../../src/smithic/rubric/default.yaml). Make `path` in `[rubric]` config actually load. |
| Where research evidence is committed | `.smithic/research.md` in the worktree | Same dir as `spec.md`. Ships in the PR so reviewers can audit what evidence drove the proposal. |

## Stage contracts (Pydantic)

Define every cross-stage value as a Pydantic model in `src/smithic/types/` so contracts are explicit. Sketch:

```
class Evidence(BaseModel):
    source: Literal["web", "reddit"]
    url: str
    title: str
    snippet: str               # ≤500 chars
    posted_at: datetime | None
    score_signal: float | None # upvotes, search rank, etc — normalized 0..1

class FeatureCandidate(BaseModel):
    title: str                 # imperative, ≤80 chars
    description: str           # 1–3 sentences
    evidence: list[Evidence]   # ≥3, ≤10
    inferred_user_pain: str    # 1 sentence

class ResearchFindings(BaseModel):
    candidates: list[FeatureCandidate]   # ≥3, ≤8
    queries_run: list[str]
    sources_used: list[str]

class AxisScore(BaseModel):
    axis: str
    score: float = Field(ge=0, le=1)
    rationale: str

class ScoredCandidate(BaseModel):
    candidate: FeatureCandidate
    axes: list[AxisScore]
    total: float
    disqualified: bool
    disqualification_reason: str | None

class ScoringResult(BaseModel):
    scored: list[ScoredCandidate]
    selected: ScoredCandidate | None       # None → abort
    abort_reason: str | None

class CriticIssue(BaseModel):
    severity: Literal["critical", "concern", "nit"]
    message: str
    file_hint: str | None

class CriticVerdict(BaseModel):
    verdict: Literal["pass", "pass-with-concerns", "revise", "abort"]
    issues: list[CriticIssue]
    spec_adherence: float = Field(ge=0, le=1)
    convention_drift: float = Field(ge=0, le=1)
    summary: str
```

These are the *minimum* fields. Add what you need; do not remove.

## Stage details

### `research.py`

Inputs: `mission: str`, `IntrospectionReport`, `ResearchConfig`, `Meter`, `auth_env`, `cli_path`.

Behavior:

1. Compose 3–5 queries derived from the mission + project type (e.g. "FastAPI rate limiting pain", "<competitor name> missing features"). Use a small Claude call to generate queries — a focused subagent with structured output. Cap at one call ≤$0.10 in API mode.
2. Fan queries across configured sources concurrently using `anyio.create_task_group`. Each source returns a list of `Evidence`.
3. Synthesize into 3–8 `FeatureCandidate` entries with another structured-output Claude call. The synthesis prompt should aggressively dedupe and require ≥3 evidence items per candidate.
4. Write `.smithic/research.md` (markdown) to the worktree for human-readability. Pydantic-encode the same data to `.smithic/research.json` for the score stage.

Failure modes:
- Zero sources reachable → abort run, surface "no research sources reachable" error in `runs.notes`.
- Synthesis returns fewer than 3 candidates → log warning, proceed with what you have.

### `score.py`

Inputs: `ResearchFindings`, `Rubric`, `IntrospectionReport`, `Meter`, auth.

Behavior:

1. Spawn a Claude session with `output_format={"type": "json_schema", "schema": ScoringResult.model_json_schema()}`.
2. System prompt: "Score each candidate against the rubric. Use the rationale field. Disqualify candidates that fail any axis ≤ `min_per_axis`. Pick the highest-total candidate that isn't disqualified."
3. Validate Pydantic. On parse failure, retry once with the validation error appended to the prompt. Second failure → abort.
4. Apply rubric thresholds — even if the model picks a candidate, recompute `total` and `disqualified` server-side. Don't trust the model's arithmetic.
5. If `selected is None` (no candidate clears thresholds), abort the run cleanly.

### `critique.py`

Inputs: spec text, `git diff <base>...HEAD` output (read via `subprocess.run` against the worktree), `IntrospectionReport`, `Meter`, auth.

Behavior:

1. **Critical**: this session must NOT inherit the implement session's context. Spawn a fresh `query()`. Do not use `resume=` or share session_id.
2. System prompt: "You are reviewing a diff for adherence to the spec. Verdict pass / pass-with-concerns / revise / abort. Map issues to spec.md sections where possible."
3. Structured output via Pydantic schema for `CriticVerdict`.
4. The orchestrator interprets:
   - `pass` → open PR normally (`draft = pr_config.draft_on_critique_concerns and False`).
   - `pass-with-concerns` → open PR with `draft=True` and `smithic-needs-review` label.
   - `revise` → call implement once more with `revise_feedback` set to the critic's `summary + issues`. Run critique again. If second verdict is anything other than `pass` or `pass-with-concerns`, abort.
   - `abort` → no PR; mark run `aborted` with `notes = critic.summary`.

### `mcp/registry.py`

Resolves the MCP server configs at runtime based on `[research].sources`. Sketch:

```
def build_mcp_servers(research: ResearchConfig) -> dict[str, McpServerConfig]:
    servers = {}
    if "web" in research.sources:
        if has_tavily_key():
            servers["tavily"] = {"type": "stdio", "command": "npx", "args": ["-y", "@tavily/mcp-server"]}
        else:
            servers["fetch"] = {"type": "stdio", "command": "uvx", "args": ["mcp-server-fetch"]}
    if "reddit" in research.sources:
        servers["reddit"] = {
            "type": "stdio",
            "command": sys.executable,
            "args": ["-m", "smithic.mcp.custom.reddit_server"],
        }
    return servers
```

Pass these to `ClaudeAgentOptions(mcp_servers=...)` only on the research stage's calls. Other stages don't need network tools.

### `reddit_server.py`

Implements an MCP server using `mcp` SDK with three tools:

- `search_subreddit(subreddit: str, query: str, limit: int = 25)` → list of posts (title, body excerpt, url, score, created_utc).
- `top_posts(subreddit: str, time: str = "week", limit: int = 25)`.
- `comments(post_id: str, limit: int = 50)` → flattened tree.

Use `httpx` against `https://www.reddit.com/r/{sub}/search.json` etc. Set `User-Agent: smithic/0.2 (https://github.com/code-monkies/Smithic)` — Reddit blocks default UAs. Cache responses for 5 minutes in-process to avoid re-hitting Reddit during a single research stage.

## Open questions for the implementing agent

These are genuinely open. Pick the option you can defend and proceed; don't block.

1. **Should the score stage see the introspection report?** Pro: better effort-fit scoring. Con: leaks repo-specific bias into market signal interpretation. Recommendation: yes, but show it as "implementation context" not "evidence."

2. **What model should the score and critique stages use?** Default Sonnet across the board. If `[auth].model` is set, use it for everything. The critique stage benefits from a stronger model since it's the safety net; consider letting `[critique].model` override.

3. **Should `--research-only` write the brief to the target repo or a separate location?** Recommendation: write to `<target>/.smithic/research-<run_id>.md` and skip worktree creation entirely. Cheap probe before committing to a build.

4. **How to handle Tavily without a key?** Recommendation: silently fall back to `mcp-server-fetch` plus a log line. Don't hard-fail and don't prompt — that breaks unattended runs.

### Manual verification

1. Inside `tests/fixtures/mock_repo/` (after `git init && git add -A && git commit -m init`):
   ```
   smithic run --config ./smithic.toml --research-only --max-usd 0.50
   ```
   Inspect the generated brief. Should propose plausible features for a tiny FastAPI app (rate limiting, /healthz, structured logging, etc.) with real Reddit/HN evidence cited.

2. Then a full run:
   ```
   smithic run --config ./smithic.toml --max-usd 5.00
   ```
   Expect a worktree, a PR, a critic verdict in `smithic status`. PR body should include the research brief excerpt and the critic summary.

3. A run where you deliberately break the implement stage (e.g. point at a repo with broken tests):
   - Critic should return `revise` once
   - Implement re-runs with feedback
   - If still broken, run aborts cleanly (no half-PR)
