# v0.3 — Swarm

> **For the implementing agent.** This plan is self-contained but assumes `v0.2` has shipped. Read [`v0.2-autonomous-ideation.md`](v0.2-autonomous-ideation.md) first if you haven't, then read the current repo state. Do not start v0.3 work if `v0.2` is incomplete — the parallel features depend on the autonomous-ideation loop existing.

## Context

`v0.1` shipped the pipeline. `v0.2` made it autonomous (research → score → critique). `v0.3` makes it parallel: a single command spawns N independent runs across N git worktrees, each producing a candidate PR. The founder wakes up to a stack of PRs to triage. This is the user-facing capability that makes "swarm" non-aspirational.

It's smaller in code than `v0.2` but more dangerous in failure modes — concurrent processes against the same target repo, shared SQLite, shared MCP servers. Most of the work is making the existing pieces safe to run N-at-a-time.

## Goals

`smithic run --runs 5 --config ./smithic.toml` must:

1. Spawn 5 fully independent runs, each in its own git worktree, each with its own branch + PR.
2. Share one local research cache so the second through fifth runs don't re-pay for queries the first one already ran.
3. Avoid catastrophic dedupe: if the rubric obviously picks the same top feature for all 5 runs (e.g. "add dark mode"), nudge subsequent runs to pick from the next-best slate instead.
4. Aggregate cost and outcomes into a parent run record so the founder sees one summary, not five.
5. Add HN and Product Hunt MCP servers so research has more breadth.
6. Polish: a real CONTRIBUTING.md walkthrough, an `examples/` directory with multiple sample missions, a short asciinema cast in the README.

## Out of scope

- **Cross-run negotiation.** Runs do not chat with each other. The "diversity nudge" (goal 3) is the closest we get; it's a one-way read of what previous runs in the same parent picked, not a coordination protocol.
- **Distributed swarms.** All runs are on one machine, in one process, against one target. Multi-machine fan-out is a future release.
- **Background scheduling.** No cron, no daemons. The user invokes; the swarm runs; the user reviews. (`smithic` is not a service.)
- **Web UI.** Still CLI-only.
- **Private-data signal sources.** Still public web + community only.

## Decisions (settled)

| Question | Answer | Why |
|---|---|---|
| Concurrency primitive | `anyio.create_task_group` with one task per run | Already async since v0.1's SDK call. No new runtime. |
| Cache backend | Chroma in embedded local mode | Lightweight, no service required, works on Windows. LanceDB is also fine if Chroma's Windows support frustrates you — they're swappable behind a thin interface. |
| Cache key | `(target_path_hash, query_text_normalized)` with a vector-similarity lookup over query embeddings | Different queries that mean the same thing should hit the same cache entry. |
| Cache TTL | Honor `[research].cache_ttl_hours` from config (default 72) | User-controllable. Always re-fetch beyond TTL. |
| Diversity strategy | Greedy: each subsequent run reads the previous runs' selected features from the parent record and asks the score stage to *prefer* a different top pick (still rubric-bound, just nudged) | Cheap, no negotiation, easy to reason about. |
| Failure isolation | One run's failure does NOT abort the parent. Failed runs are logged; siblings keep running. | "Some PRs better than no PRs" is the value proposition. |
| Concurrent SQLite writes | Each run opens its own short-lived connection (the v0.1 `Memory` class already does this) + WAL mode | SQLite WAL handles concurrent readers + one writer fine for our write rates. |
| Worktree cleanup | Default: never automatic. `smithic clean --all` removes everything Smithic owns; `smithic clean --keep-failed` removes only worktrees from completed runs. | Founder might want to inspect the failed ones. |
| HN MCP | Custom in-tree, Algolia API (`https://hn.algolia.com/api/v1/search`) | Free, no auth, well-documented. ~120 LOC. |
| Product Hunt MCP | Custom in-tree, GraphQL, requires user-supplied PH API token | Optional source — if no token, skip silently with one log line. |

## Architecture overview

```
parent run (CLI invocation)
├── run 1 (worktree 1, branch 1, PR 1)
├── run 2 (worktree 2, branch 2, PR 2)
├── ...
└── run N (worktree N, branch N, PR N)
        └── all share: research cache, MCP servers, parent run record
```

The parent is a thin coordinator. Each child run is the v0.2 single-run flow, with two additions:

1. **Cache hook in `research.py`**: before any source query, ask the cache. After any source query, write to the cache.
2. **Diversity hook in `score.py`**: before scoring, read the parent's `selected_features_so_far` list (from the DB) and pass it into the scoring prompt as "candidates already claimed by sibling runs — prefer something different if rubric scores are close."

## Concurrency model

```
async def run_swarm(config, config_dir, feature, db_path, n_runs, ...) -> list[RunOutcome]:
    parent_id = new_run_id()
    memory.start_parent_run(parent_id, target_path, n_runs)
    cache = ResearchCache(db_path.parent / "cache")

    outcomes: list[RunOutcome] = []
    async with anyio.create_task_group() as tg:
        for i in range(n_runs):
            tg.start_soon(_run_child, config, config_dir, feature, db_path,
                          parent_id, cache, outcomes)

    memory.finish_parent_run(parent_id, outcomes)
    return outcomes
```

`_run_child` calls the existing `run_once` with `parent_run_id=parent_id, cache=cache`. Each child catches its own exceptions and appends a `RunOutcome` (success or failure) — exceptions never propagate out of a child task, otherwise one bad run kills the parent.

### Worktree race

Two child runs starting near-simultaneously could both call `git fetch origin main` at the same time. Git tolerates this in most cases but it's noisy. Add an `anyio.Lock` per target repo path, held only across the `git fetch` + `git worktree add` operations. Drop the lock before the long-running implement stage.

### SQLite race

Each `Memory.<method>()` opens a fresh connection (already true in v0.1). Switch the connection setup to:

```
conn = sqlite3.connect(self.db_path, isolation_level=None, timeout=30.0)
conn.execute("PRAGMA journal_mode=WAL")
conn.execute("PRAGMA busy_timeout=30000")
```

WAL mode allows concurrent readers and one writer; the timeout backs off on contention. Don't try to share a connection across tasks — that's where the real bugs are.

### MCP server race

MCP servers are spawned per `query()` call by the SDK. Multiple child runs each spawn their own — they don't share processes. The only contention is the Reddit/HN endpoints rate-limiting us. Keep the in-process 5-minute response cache from v0.2's Reddit server and add the same to HN.

## Cache schema

Chroma collection `smithic_research`, each entry:

```
id: str                 # sha256(target_path | normalized_query)
embedding: vector       # of normalized_query
metadata:
  target_path_hash: str
  source: "web" | "reddit" | "hn" | "producthunt"
  query: str
  fetched_at: iso datetime
  cache_ttl_hours: int
document: str           # JSON-encoded list of Evidence
```

Lookup:
1. Normalize the query (lowercase, strip whitespace).
2. Embed it (use the SDK's small embedding model or a local one — `sentence-transformers/all-MiniLM-L6-v2` if you want to keep it offline; document the choice).
3. Similarity search with `where={"target_path_hash": ...}` and similarity threshold ≥ 0.85.
4. If hit and `now - fetched_at < cache_ttl_hours`, return cached evidence.
5. Otherwise, miss → run the source query, write the result back.

Cache writes are best-effort — a write failure should warn but not fail the run.

## Diversity strategy

When a child run starts the score stage, the orchestrator passes `previously_selected: list[str]` from the parent record (each entry is a candidate `title`). The scoring prompt appends:

> "Sibling runs in this swarm have already selected: <titles>. If a candidate's total score is within 0.05 of the top, prefer the one most different from the already-selected list. The rubric still wins ties — never pick a disqualified candidate."

This is a soft nudge, not a hard rule. The rubric's `min_total` and `min_per_axis` thresholds always apply. If all non-claimed candidates fail the threshold, the run aborts cleanly (better no PR than a duplicate).

## Open questions for the implementing agent

1. **What if the user passes `--runs 1`?** Recommendation: short-circuit to `orchestrator.run_once` directly — no parent record, no cache. Saves complexity for the common case. Cache only matters when N > 1 in the same invocation.

2. **Should the cache survive across `smithic` invocations?** Pro: parent A's cache helps parent B. Con: stale data, harder to reason about. Recommendation: yes, persist on disk (`<target>/.smithic/cache/`); honor TTL; document that `smithic clean --cache` exists for a fresh start.

3. **What embedding model?** Trade-off: sentence-transformers is offline (good for local-first) but adds a heavy dependency. The Anthropic SDK doesn't ship an embedding API. Recommendation: ship with `sentence-transformers` as an optional extra (`pip install smithic[cache]`) and skip caching with a clear log line when it isn't installed. Don't make a 200MB model a hard dep.

4. **How verbose should the parent's CLI summary be?** Recommendation: a compact table by default (run_id, status, branch, PR url, $cost, notes). `--verbose` adds the implement summary excerpts. Avoid Rich livedraws — they fight with subprocess output.

5. **Should we cap `--runs`?** Hard cap at 20 in v0.1's `SwarmConfig.parallel_runs` Pydantic schema (`le=20`). Keep that; document why (Reddit/HN rate limits start mattering past 5–10 concurrent fetches against the same target).

### Integration test (`SMITHIC_LIVE=1` only)

Three concurrent runs against `tests/fixtures/mock_repo/`:

1. All three open separate PRs.
2. The second and third runs hit the cache for at least one query (verify via cache stats logged).
3. The three selected features are not identical (the diversity nudge worked, OR rubric naturally picked different things).
4. One run failing doesn't kill the others — simulate by deliberately constraining the second run's budget low.

### Manual verification

1. `smithic run --config ./smithic.toml --runs 3 --max-usd 2.00` against a real target.
2. Watch three branches pushed and three PRs opened.
3. `smithic status` shows the parent and all children.
4. Run again 10 minutes later — verify cache hit messages in the second invocation.
5. `smithic clean --keep-failed` after a partial-success swarm; failed worktrees remain, succeeded ones removed.

## Failure-mode reminders

The 41–86.7% production failure rate of multi-agent systems is dominated by coordination, not capability. Concrete things to verify before marking this release done:

- A run's worktree is never deleted automatically; the founder always has the artifact.
- The cache never silently returns stale data past TTL.
- A rate-limited Reddit/HN call doesn't kill the run — it returns whatever evidence it has and logs the partial degradation.
- `smithic status` is the source of truth — if the DB says a run failed, it actually failed and didn't silently push a bad PR.
- All paths are `pathlib.Path`; all subprocess calls are `shell=False`. (Re-check this after any `os.system`/`shell=True` regressions sneak in.)
- The label blocklist from [`config.py`](../../src/smithic/config.py) `RESERVED_LABELS` is honored on every PR, including parent-aggregated PRs.

If any of those slip, fix them before shipping.
