# Mission

Sample mission file. Replace this with your project's actual mission so Smithic
has something concrete to ground feature proposals against.

A good mission file answers four questions in 1–2 paragraphs:

1. **What is this project for?** What does it do that nothing else does, or does
   better than the alternatives?
2. **Who is it for?** Be specific — "developers" is not a user segment; "Python
   data engineers maintaining ETL pipelines on AWS" is.
3. **What does success look like?** What metric or outcome would tell you the
   project is winning?
4. **What is explicitly out of scope?** This is the most undervalued half of the
   document. It saves Smithic (and your future self) from building the wrong
   thing.

The shorter and sharper, the better. One page is plenty.
